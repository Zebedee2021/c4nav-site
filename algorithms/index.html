    算法¶

    c4nav-core 已实现的算法和可扩展方向。

    已实现：DQN 系列（4 种变体）¶

    全部 4 种变体在同一个 DQNAgent 类中实现，通过配置开关区分：

    算法 配置文件 关键开关
    DQN (基线) baseline.yaml 全部关闭，MSE 损失
    Double DQN ddqn.yaml use_double_dqn: true
    D3QN d3qn.yaml + network_type: dueling
    ID3QN (本文) improved_d3qn.yaml + PER + 软更新 + 奖励塑形 + Huber + 学习率调度

    可扩展：通过 SB3 接入 PPO¶

    环境遵循标准 Gymnasium 接口，添加 PPO 仅需约 20 行封装代码。详见 PPO 扩展。

    算法-环境解耦¶

    环境与算法完全解耦：

    • 环境定义 state_dim 和 action_dim
    • 智能体在构造函数中接收这些参数
    • 修改观测维度无需改动任何智能体代码
    # 环境侧：state_dim 由配置自动计算
self.state_dim = num_rays + 4  # 180 + 4 = 184

# 智能体侧：接收任意 state_dim，网络自动适配
agent = DQNAgent(state_dim=env.state_dim, action_dim=env.action_dim, ...)
# 首层自动变为 Linear(state_dim, 256) -- 无需手动修改

    实际意义：更换 LiDAR 型号¶

    比如把 LiDAR 从 180 射线换成 360 射线，只需改 YAML 配置：

    lidar:
  num_rays: 360    # 原来是 180
  max_range: 18.0
  fov_degrees: 360.0

    环境输出的观测维度从 184D 自动变为 364D，智能体网络首层从 Linear(184, 256) 自动变为 Linear(364, 256)。零行代码改动。

    代码不改，但需要重新训练¶

    "不改代码"不等于"结果一样"。不同传感器配置下，策略需要重新学习：

    LiDAR 变化 对策略的影响
    射线数 180→360 观测更密，分辨率更高，但输入维度翻倍
    射线数 180→60 观测更稀疏，可能漏掉小障碍物
    量程 18m→50m 更早发现障碍物，策略可能更保守
    FOV 360°→120° 后方盲区，需要学会不后退

    旧模型权重维度不匹配，无法直接加载，需要重新训练。但训练脚本和智能体代码本身不需要任何修改。

    类比理解

    就像换了一副不同度数的眼镜：眼镜框（代码接口）兼容任何镜片，但换了镜片后大脑（策略网络）需要重新适应。适应过程（重新训练）用的是同一套学习方法（DQN），不需要换方法。

    这就是"解耦"的工程价值：环境的物理参数随便改，算法侧一行代码不动，只需重新训练。 对于研究来说，这意味着可以快速做不同传感器配置的对比实验。
